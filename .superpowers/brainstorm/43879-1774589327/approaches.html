Implementation Approach

Ember ダッシュボードのスタイル基盤をどうするか

A

CSS Custom Properties に移行

cogmem と同じ方式。Tailwind を剥がして CSS 変数ベースに。

Pros

  • cogmem と完全に同じ構造 — 将来の共通化が容易
  • デザイントークンの一元管理
  • ダークモード切替も変数だけで可能

Cons

  • 全コンポーネントの className 書き直し（大工事）
  • Tailwind のユーティリティ（flex, grid, gap等）も失う
  • 工数が最も大きい
B

Tailwind テーマカスタマイズ

tailwind.config.js で Ember Glow のカラートークンを定義。既存クラスの色名を差し替え。

Pros

  • Tailwind のユーティリティはそのまま使える
  • 色名の置換で済む（gray-900 → ember-bg 等）
  • 工数が中程度

Cons

  • cogmem との構造的統一はされない
  • 全コンポーネントの色クラスを grep & replace
  • タイポグラフィ・spacing の統一は手作業
C

ハイブリッド: CSS 変数 + Tailwind

CSS 変数でカラートークンを定義し、Tailwind の arbitrary value [var(--xxx)] で参照。レイアウトユーティリティは Tailwind のまま。

Pros

  • カラートークンは cogmem と同じ構造
  • flex, grid, gap, p-*, m-* 等のレイアウトは維持
  • 将来の共通デザインシステムへの移行パスが明確
  • 変更箇所が最小限

Cons

  • bg-gray-800 → bg-[var(--surface)] への書き換えは必要
  • 2つのシステムが混在する過渡期がある

推奨: C. ハイブリッド — カラーの統一感を得つつ、Tailwind のレイアウト機能を維持。工数と効果のバランスが最も良い。
