### [DECISION] ウィンドウ駆動の設計判断
- 旧ウィンドウフラッシュ後の anchor は `pending_buffer_len` を持ち越す（旧窓の20件分が新窓に滲まないように）
- 新窓に新スニペットが0件の場合は `_start_or_update_meeting_digest_batch` が None を返す（空送信防止）
- 窓遷移時は `skip_update=True` で `_start_or_update_meeting_digest_batch` を呼ばずに pending state そのまま送信
- gcal cache TTL は 300s 据え置き、ただしキャッシュ済みイベントが既に終了している場合は強制再フェッチ
- `MEETING_SUMMARY_BATCH_SEC` は安全網（end_ts 取得失敗時の上限）として残す、削除はしない

### [INSIGHT] テスト追加で検出した境界バグ
- 当初 `_clear_meeting_digest_batch` は anchor 持ち越しなしで実装 → `test_maybe_send_meeting_digest_posts_to_slack_once` が 2 回送信する regression
- 既存テストは COOLDOWN 経路を暗黙に依存していた（同 transcript → 同 signature → cooldown でブロック）
- window_key を signature に混ぜたことで signature 不一致 → cooldown 抜けてしまう副作用が発生
- → anchor 持ち越し + `transcript.strip()` 空チェックで早期 None return することで解決
