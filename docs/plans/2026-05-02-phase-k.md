## K1: Reactivity リセットバグ根本修正

### 現状

`settings.json` の `ambient_reactivity` が 5 → 4 に意図せず戻る。

**競合経路の特定（コード読み取り結果）:**

1. `ImproveLoopPanel.tsx:89` と `DebugPanel.tsx:44` が `useState(4)` でハードコードされた初期値を持つ
2. パネルが開くたびに `GET /api/settings` → `setReactivity(r)` で上書きされるが、WS 接続後すぐに `update_settings` が飛ぶ場合は初期値 4 が先に書き込まれる可能性がある
3. `useEmberChat.ts:266` の `sync_settings` ハンドラは `Date.now() - lastSaveTimeRef.current < 500` で 500ms guard を持つが、パネルが開く前の接続時 sync は guard をすり抜ける
4. `app.py:6673`: WS `update_settings` は `_settings.update(data.get("settings", {}))` でフルマージ → フロントエンドが `ambient_reactivity` を誤って含む `settings` オブジェクトを送ると上書きされる

**根本原因:** `updateSetting()` が `{ type: 'update_settings', settings: next }` でオブジェクト全体を送るため、`ambient_reactivity` を含む `_settings` がパネルの `useState(4)` デフォルト値で上書きされる。

### 変更内容

**Step 1: `app.py` — WS `update_settings` から `ambient_reactivity` を除外**

`app.py:6673` の `update_settings` ハンドラで、`ambient_reactivity` キーをマージ対象から除外する。
専用エンドポイント `/api/ambient/reactivity` 経由のみ変更を受け付ける。

```python
# app.py:6673 付近
incoming = data.get("settings", {})
# ambient_reactivity は専用エンドポイント経由のみ変更可
incoming.pop("ambient_reactivity", None)
_settings.update(incoming)
```

**Step 2: `ImproveLoopPanel.tsx` と `DebugPanel.tsx` — 初期値を null に変更**

```tsx
// Before
const [reactivity, setReactivity] = useState(4);
// After
const [reactivity, setReactivity] = useState<number | null>(null);
```

`handleReactivityChange` と表示部分に null チェックを追加。サーバーから取得完了前は select を disabled にする。

**Step 3: `handleReactivityChange` を専用エンドポイントのみ使用に統一確認**

`ImproveLoopPanel.tsx:191` と `DebugPanel.tsx` の `handleReactivityChange` は既に `/api/ambient/reactivity` を呼んでいる（`updateSetting` を経由しない）。WS の `update_settings` には含まれないことを確認済み。

### 影響範囲

- `app.py` WS ハンドラ（1行除外）
- `ImproveLoopPanel.tsx`（初期値 + null チェック）
- `DebugPanel.tsx`（同上）

### テスト戦略

1. パネルを開く → reactivity が 5 に設定済みの状態で他の設定変更（voiceSelect 等）を行う → reactivity が 5 のまま維持されることを確認
2. `GET /api/settings` レスポンスに `ambient_reactivity: 5` が含まれることを確認
3. WS の `update_settings` に `ambient_reactivity` が含まれる payload を送っても変化しないことを curl で確認

## K2: is_meeting 誤判定対策

### 現状

PIVOT 等の YouTube ビジネス系動画を `is_meeting=True` と誤判定 → H3 が reply → backchannel に降格 → 有益情報が抑制される。

**現在の判定フロー（`app.py:1477`）:**
`_build_context_summary()` が 5 分ごとに transcript + `_media_ctx.inferred_type` を LLM に投げ、`is_meeting` boolean を返す。media_hint として `inferred_type` は注入されているが、`youtube_talk` の場合でも LLM が `is_meeting=True` を誤って返す場合がある。

### 変更内容

**Step 1: YouTube 動画タブ検出による is_meeting=False 強制**

`_build_context_summary()` の `result` 処理部（`app.py:1477`）に後処理バリデーションを追加。

```python
# app.py:1477 付近、is_meeting 設定箇所の直後
_raw_is_meeting = bool(result.get("is_meeting"))
# K2: youtube_talk/youtube_etc/vtuber/anime/music 系は is_meeting=False を強制
_MEDIA_SUPPRESS_MEETING = {"youtube_talk", "youtube_etc", "vtuber", "anime", "music", "drama", "news"}
if (
    _raw_is_meeting
    and _mc.inferred_type in _MEDIA_SUPPRESS_MEETING
    and _mc.confidence >= 0.6
):
    logger.info(
        f"[context_summary] K2 override: is_meeting=True suppressed "
        f"(inferred_type={_mc.inferred_type}, conf={_mc.confidence:.2f})"
    )
    _raw_is_meeting = False
_context_summary.is_meeting = _raw_is_meeting
```

**Step 2: カレンダーフォールバック（`gcalcli` 活用）**

既存の `mcp__claude_ai_Google_Calendar` MCP はリアルタイム非同期呼び出しには不向き。代わりに `gcalcli` をサブプロセスで呼んで「現在時刻が会議ブロック内か」を確認する軽量チェックを実装。

```python
async def _check_calendar_meeting_now() -> bool | None:
    """gcalcli で現在時刻が会議ブロック内か確認。失敗時は None（無視）。"""
    try:
        now_str = datetime.now().strftime("%Y-%m-%d")
        proc = await asyncio.create_subprocess_exec(
            "gcalcli", "agenda", "--nocolor", "--nodeclined",
            "--tsv", now_str, now_str,
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.DEVNULL,
        )
        stdout, _ = await asyncio.wait_for(proc.communicate(), timeout=5.0)
        lines = stdout.decode().strip().split("\n")
        now_ts = time.time()
        for line in lines:
            parts = line.split("\t")
            if len(parts) >= 4:
                # TSV: date, start, end, title
                try:
                    start = datetime.strptime(f"{parts[0]} {parts[1]}", "%Y-%m-%d %H:%M").timestamp()
                    end = datetime.strptime(f"{parts[0]} {parts[2]}", "%Y-%m-%d %H:%M").timestamp()
                    if start <= now_ts <= end:
                        return True
                except ValueError:
                    continue
        return False
    except Exception:
        return None
```

カレンダーチェックは `_build_context_summary()` 内でオプションとして実行（タイムアウト=5s、失敗無視）。

**Step 3: Akiraさん発話比率による補強**

`AmbientListener._recent_speakers` は既に直近60秒の話者ログを保持している。`app.py` で `record_speaker()` を呼ぶ際に speaker 名も記録されている（`app.py:6001`）。

`_ambient_listener._recent_speakers` から Akiraさん（`current_speaker == "Akira"` 相当）の発話比率を計算するユーティリティを `AmbientListener` に追加:

```python
# ambient_listener.py に追加
def akira_speech_ratio(self, window_sec: int = 300) -> float:
    """直近 window_sec 内の発話のうち Akiraさんの割合 (0.0-1.0)。データ不足時は -1.0。"""
    now = time.time()
    recent = [s for s in self._recent_speakers if now - s["ts"] < window_sec]
    if len(recent) < 5:  # データ不足
        return -1.0
    akira_count = sum(1 for s in recent if s.get("speaker") == "Akira")
    return akira_count / len(recent)
```

`_build_context_summary()` でこの比率を `media_hint` に付加してLLMに渡す。会議なら30%以上、独り視聴なら<10%という目安をプロンプトに明示。

**Step 4: LLM プロンプト hint 注入統合**

`_build_context_summary()` の `media_hint` 構築部に、カレンダー結果と発話比率を追記:

```python
# app.py の _build_context_summary() 内
calendar_hint = ""
cal_result = await _check_calendar_meeting_now()
if cal_result is True:
    calendar_hint = "\n[カレンダー] 現在、Google Calendar の会議ブロック内です（is_meeting=True の根拠を強化）"
elif cal_result is False:
    calendar_hint = "\n[カレンダー] 現在、Google Calendar に会議ブロックはありません（is_meeting=False の根拠）"

akira_ratio = _ambient_listener.akira_speech_ratio() if _ambient_listener else -1.0
akira_hint = ""
if akira_ratio >= 0:
    akira_hint = (
        f"\n[発話比率] 直近5分の Akiraさん発話比率: {akira_ratio:.0%}。"
        "（会議の目安: 30%以上、独り視聴/作業: 10%未満）"
    )
```

## K3: 会議補助モード（reactivity=5 + is_meeting=True）

### 現状

`ambient_listener.py:300-303` の H3 ロジック:

```python
if (context_summary.is_meeting
        and context_summary.confidence >= AMBIENT_MEETING_SUPPRESS_CONF):
    return "backchannel"
```

`reactivity=5` でも `is_meeting=True` なら reply → backchannel に降格される。
会議中に Mei にイヤフォン経由で補助を求めている場合に有益情報が抑制される。

### 変更内容

**Step 1: `AmbientListener.decide_intervention()` に会議補助モード分岐を追加**

`ambient_listener.py:300` の H3 ブロックを以下に置き換え:

```python
if context_summary is not None and not context_summary.is_stale():
    if (context_summary.is_meeting
            and context_summary.confidence >= AMBIENT_MEETING_SUPPRESS_CONF):
        # K3: reactivity=5 の場合は会議補助モード（suppression を反転）
        if self.effective_reactivity >= 5:
            pass  # suppress しない（後続の通常判定に fallthrough）
        else:
            return "backchannel"
    if (context_summary.activity == "idle"
            and context_summary.confidence >= AMBIENT_IDLE_SUPPRESS_CONF):
        return "skip"
```

**Step 2: intervention ラベル `meeting_assist` を返す**

`decide_intervention()` の戻り値に `meeting_assist` を追加。呼び出し元（`app.py` の ambient 処理パス）で分岐:

```python
# decide_intervention の末尾、H3 fall-through 後
if (result == "reply"
        and context_summary is not None
        and not context_summary.is_stale()
        and context_summary.is_meeting
        and self.effective_reactivity >= 5):
    return "meeting_assist"
```

**Step 3: `app.py` の ambient 処理パスで `meeting_assist` を処理**

`decide_intervention()` の呼び出し元（ambient 処理ループ内）で `meeting_assist` を受けた場合の LLM プロンプト切り替えを実装。

`build_llm_prompt()` に `mode` 引数を追加:

**Step 4: タイミング制約（後発話間隔 3秒以上）**

`app.py` の ambient 処理パスで `meeting_assist` の場合に最後の Mei 発話から 3 秒以上経過していることを確認してから TTS を実行。

### K2/K3 同時設計の理由

K3 の会議補助モードは K2 の `is_meeting` フラグ信頼度が高くないと誤発動する。K2 を先に ship し、is_meeting の精度を確認してから K3 を有効にする順序が安全。ただし仕様は同時に設計しておくことで、K2 実装時に K3 の前提（`_MEDIA_SUPPRESS_MEETING` の対象範囲、カレンダー hint の粒度等）を考慮できる。
